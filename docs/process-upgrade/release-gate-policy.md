# 发布门禁与退出准则（编码后测试阶段）

本策略用于控制代码从“开发完成”进入“可发布状态”的准入和退出标准。

## 1. 适用范围

- 所有功能变更、缺陷修复、重构类改动。
- 核心链路、鉴权、支付、数据处理等敏感模块适用加严规则。

## 2. 门禁维度

### 2.1 静态质量门禁（必过）

- TypeScript类型检查通过。
- 自动化测试通过（单元/集成）。
- 依赖漏洞扫描通过（高危与严重漏洞为0）。

### 2.2 测试质量门禁（必过）

- 测试报告完整并关联需求ID。
- 核心模块覆盖率满足团队基线（建议 >= 80%，可按模块分级）。
- Sev1/Sev2缺陷必须关闭。
- Sev3缺陷需有明确处置计划与负责人。

### 2.3 发布风险门禁（必过）

- 回滚方案可执行且经过演练或评审。
- 监控、日志、告警项已配置并验证。
- 涉及数据迁移时，提供回滚与数据修复预案。

## 3. 退出准则（Go/No-Go）

仅当以下条件全部满足时可发布：

1. CI门禁全部通过。
2. 测试报告结论为“发布”或“有条件发布且条件已满足”。
3. 风险项已关闭或获批准豁免（含到期时间与责任人）。

出现以下任一情况必须阻断发布：

- 存在未关闭Sev1/Sev2缺陷。
- 高危/严重依赖漏洞未处置。
- 回滚方案缺失或不可执行。

## 4. 豁免流程

- 仅允许短期、有时限、可追踪的豁免。
- 豁免必须包含：问题描述、风险评估、缓解措施、到期时间、审批人。
- 到期自动进入阻断状态，需重新评审。
